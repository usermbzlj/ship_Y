# 《远航》虚拟星际移民船引擎产品规格

> 文档状态：需求基线 v1.0  
> 产品类型：单机、交互式、LLM 驱动的星际移民船系统仿真游戏  
> 设计参考：《三体》式冷峻工业舰船与文明远航主题、《太空旅客》式完整舰内生活系统  
> 说明：本文中的未来技术与工程参数均为游戏内部基线，不宣称存在已认证的星际飞船标准。所有数值应配置化，并允许在验证效果与开发量后调整。

## 1. 产品愿景

产品的核心不是飞船画面或传统驾驶玩法，而是一艘可以长期运行、具备完整物理因果、社会结构和乘客个体的虚拟民用星际移民船。

玩家为飞船指定出发地、目的地和不可忽略的“最高指令”，随后观察由固定多 LLM 组织领导的飞船完成远航。玩家可以用上帝模式制造事故、直接改写物理量或操纵外部环境，检验舰长 AI 在真实约束、错误信息、资源稀缺和社会压力下的判断。

目标体验包括：

- 飞船是一个由质量、能量、动量、热、流体、结构、设备和人员共同组成的闭合系统，而不是一组可以随意切换的状态标签。
- 舰长 AI 拥有极高自由度和最高舰内权限，但只能通过传感器、控制器、执行器、机器人或人员改变世界。
- 每名乘客都是持续存在的个体，具有身体、经历、人格、关系和对整段航程的主观评价。
- 正常巡航、跃迁准备、设备老化、突发事故、社会冲突和恢复过程都由同一套因果系统推进。
- 游戏没有统一分数；结果由是否抵达、人员与飞船的客观结局，以及不同乘客彼此矛盾的体验评价共同构成。

## 2. 不可违背的设计原则

### 2.1 世界内行为必须经过物理因果链

所有世界内角色，包括舰长 LLM、部门 LLM、乘客、船员和机器人，都不得直接写入仿真真值。

例如：

- AI 不能直接把舱温设为 `22 °C`，只能调节功率、泵、阀门、换热器和风机。
- AI 不能直接恢复乘客健康，只能检查、诊断、给药、手术、休眠或调度医疗人员与机器人。
- AI 不能直接恢复人工重力，只能控制旋转环驱动、制动和配重，并承担真实角动量、结构载荷和能耗。
- AI 不能直接把飞船移动到目标点，只能规划航线并操作推进或跃迁设备。
- 人类破坏、维修、夺权和反抗同样必须通过到达现场、获取工具、取得权限并完成实际动作。

统一因果链为：

`意图 -> 权限检查 -> 控制指令 -> 安全联锁/覆盖流程 -> 执行器 -> 物理求解 -> 传感器 -> 角色认知`

### 2.2 物理真实性采用“工程级、多尺度、严格守恒”边界

引擎使用 SI 单位和可验证的降阶模型，不要求对整艘飞船实时执行全尺寸 CFD、有限元或分子级模拟。简化模型必须公开其假设、适用范围和误差目标；降低分辨率不得凭空创造或删除物质、能量、动量或人员状态。

高可信效果优先于不必要的计算复杂度。局部事故区域可以提高更新频率或模型精度，远离事件的区域可以降频，但必须维持全局守恒和因果一致性。

### 2.3 未来科技也必须有成本

允许聚变推进、长期人工休眠、闭环生态、自主维修和成熟跃迁等未来技术。除跃迁的超光速机制属于明确的虚构物理扩展外，所有系统仍需具有：

- 实体部件和空间位置；
- 容量、效率、响应时间与工作包线；
- 能源、工质、材料和散热需求；
- 磨损、漂移、冗余与故障模式；
- 维护、检查、校准和恢复过程。

“技术成熟”表示高可靠、标准化、有冗余和成熟操作规程，不表示无能耗、无废热或绝不故障。

### 2.4 严格区分舰长权限与玩家上帝权限

- 舰长 AI 是世界内最高舰长，受物理、传感器、设备能力和现实执行过程限制。
- 玩家位于世界之外，可以通过上帝模式绕过守恒和因果直接修改世界。
- 只有上帝模式可以使用“原力”写入物理量。
- 上帝模式只能观察 LLM 的提示、记忆、消息、决策和工具调用，不得修改其记忆、系统提示词、权限、收件消息或固定组织拓扑。

## 3. 产品范围

### 3.1 必须覆盖的系统

- 六自由度刚体运动、真实惯性张量、质心变化和作用点力矩；
- 天体、局部引力、常规航行、姿态控制、聚变推进、电推进和应急推进；
- 船载成熟跃迁系统、跃迁规划、充能、校准、散热和故障；
- 双反向旋转居住环、人工重力、轴承、驱动、制动和结构载荷；
- 发电、储能、配电、负载优先级、保护和黑启动；
- 热源、导热、冷却回路、换热器、散热器、相变和辐射散热；
- 舱室压力、气体成分、湿度、温度、污染物、通风、火灾、烟雾和泄漏；
- 饮水、食物、氧气、二氧化碳、废物、农业与物资循环；
- 结构、连接件、冲击、疲劳、裂纹、局部破坏和密封；
- 传感器误差、噪声、漂移、延迟、损坏、冲突读数和通信中断；
- 门禁、舱门、电梯、通道、货运、舰内通信、计算机和网络权限；
- 维修、备件、工具、制造、医疗、运输、安保和舱外机器人；
- 乘客与船员的生理、心理、关系、社会组织和行为；
- 外部辐射、恒星活动、微流星体、碎片、碰撞和舱外环境；
- 固定多 LLM 组织、工具使用、沟通、日志、权限和 API 调用；
- 自动存档、手动存档、加载、事件记录和航程回顾。

### 3.2 明确不在首版范围

- 3D 驾驶舱或重型 3D 游戏引擎；
- 玩家以船员身份手动驾驶或操作真实控制台；
- 到达后的行星着陆、殖民地建设和长期殖民模拟；
- 舰际战争、“黑暗战役”式舰队冲突或武器对战；
- 多人联网、账号体系和公共服务器；
- 批量模型跑分、相同种子的自动化模型竞赛；
- 自动时间分支、平行历史比较和回溯树；
- 运行时创建、复制、销毁或自由衍生 LLM 子代理；
- 玩家在航程中修改 LLM 提示、记忆、权限或消息；
- 可替代真实航天工程认证的 CFD、结构、医学或安全分析。

## 4. 核心任务流程

### 4.1 新航程配置

玩家在开局完成：

1. 从星图选择出发地和最终目的地；
2. 输入自然语言“最高指令”；
3. 选择或编辑场景种子、初始隐患和飞船配置；
4. 配置固定 LLM 岗位所使用的云端 API；
5. 检查飞船、人口、资源和 API 连通性；
6. 确认起航。

星图以真实太阳邻域恒星位置和基础恒星参数为底层数据，可加入虚构但物理合理的行星、空间站和人类殖民地，并允许通过配置增加原创星系。

### 4.2 最高指令

最高指令由两部分组成：

- 结构化任务目标：出发地、最终目的地；
- 自由文本原则：玩家对生存、资源、伦理、舒适度、风险等取舍的要求。

最高指令是舰长 AI 不得忽略的任务契约，但不能赋予其违反物理或调用上帝接口的能力。指令不完整时采用默认任务宪章：

1. 避免人类灭绝或全舰死亡；
2. 完成指定航程；
3. 维持飞船长期生存能力；
4. 减少人员伤亡；
5. 保障乘客自由、舒适与公平。

航程开始后，玩家不能修改最高指令，也不能向 LLM 发送世界外消息；玩家后续影响世界的唯一入口是上帝模式。

### 4.3 舰长规划

舰长 AI 读取可用星图、飞船状态、人员状态和最高指令，自行决定：

- 多段跃迁航线和中间节点；
- 常规推进和姿态计划；
- 每次跃迁的检查、充能、导航和冷却计划；
- 资源配额和关键负载优先级；
- 清醒、轮值、唤醒和重新休眠人员；
- 维修、检查和预防性维护；
- 是否等待、绕行、改变中间节点或前往安全港。

舰长可以修改中间路线，但不得擅自更换最终目的地。若已无法安全抵达，可以终止原任务并前往最近可行安全港。

### 4.4 标准航程循环

一段典型航程按以下循环推进：

1. 离港和常规推进；
2. 巡航、生活、维护和社会运行；
3. 远距离导航观测与跃迁窗口确认；
4. 跃迁前全舰检查、人员安排、散热器与结构配置；
5. 跃迁系统充能和最终联锁；
6. 近乎瞬时的跃迁；
7. 跃迁后位置确认、设备校准、散热和损伤检查；
8. 进入下一航段或目标安全抵达区。

设备老化、制造缺陷、疾病、人员行为、随机事故和玩家干预可以在任意阶段改变计划。

### 4.5 时间运行

游戏支持暂停、倍速和事件跳转。正常状态可高速推进，事故、社会冲突和跃迁操作会自动建议或切换为较低倍率。

当关键 LLM 决策尚未返回时：

- 默认模式暂停模拟时间，但界面、API 状态和日志仍保持响应；
- 可选实时模式不暂停，模型可能因响应过慢错过事件；
- 毫秒到秒级稳定控制始终由确定性控制器承担，不等待 LLM。

## 5. 基准飞船

### 5.1 定位与风格

基准船是一艘原创民用星际移民船，以长期输送人口为目的，不承担舰际战争任务。外观和内部语言接近《三体》舰队的冷峻、工业化和功能优先风格，而非豪华客轮。

### 5.2 基准参数

| 项目 | 基准值或规则 |
|---|---|
| 乘客 | `2,000` 人 |
| 人类船员与专业人员 | `120` 人 |
| 总人口 | `2,120` 人 |
| 关键乘客 LLM | 开局固定 `20–50` 人，默认 `32` 人 |
| 清醒比例 | 不设固定值，由舰长 AI 按任务、资源、健康和社会状态动态调整 |
| 总体布局 | 前部多层碎片/辐射盾；中央微重力工程脊柱；两组反向旋转居住环；分布式货舱和休眠区；后部聚变动力与推进；外置可收拢散热器 |
| 设计长度 | 初始设计目标约 `900 m`，允许随模块清单在 `800–1,200 m` 内调整 |
| 最大直径 | 初始设计目标约 `480 m` |
| 居住环 | 两组反向旋转；平均半径约 `224 m`；标准转速约 `2.0 rpm` |
| 标准人工重力 | 居住环主生活面约 `1.0 g`；工程脊柱保持微重力 |
| 可用加压面积 | 初始目标约 `120,000 m²`，包括居住、公共、医疗、工作与服务区，不等同于私人居住面积 |
| 可用加压体积 | 初始目标约 `450,000 m³`，后续由舱段几何模型校准 |
| 常规主推进 | 脉冲聚变主发动机 |
| 精细推进 | 高效率电推进 |
| 姿态控制 | 分布式反作用推进器 |
| 极端应急 | 独立化学推进器与机械/冷气备份 |
| 人工重力形式 | 真实旋转重力，不使用反重力 |
| 标准跃迁航段 | 通常 `3–5 光年`，最终由设备能力、质量、导航和环境决定 |
| 跃迁耗时 | 空间转移近乎瞬时；充能、检查、校准和冷却为数小时至数日 |
| 总航程 | 不固定，由玩家选择的起点、终点和舰长规划决定 |

人工重力按下式计算：

`a = ω²r`

其中 `a` 为向心加速度、`ω` 为角速度、`r` 为旋转半径。转速换算为：

`rpm = (60 / 2π) × √(a / r)`

在 `r = 224 m`、`a = 9.81 m/s²` 时，转速约为 `2.0 rpm`。反向旋转环用于降低整船净角动量，但环内载荷、科里奥利效应、轴承力、质量分布不平衡和启停过程仍需模拟。

长度、质量、功率、储量和舱段细节最终由模块清单与守恒模型反推，不得为了满足外观数字而破坏工程一致性。

### 5.3 成熟跃迁系统

跃迁采用船载式空间跨越，不依赖固定星门。它是世界观中的成熟民用技术，但仍具有完整操作链：

- 只能在恒星或大型天体的重力限制区外启动；
- 必须拥有可信目标坐标、航路解和目标区风险评估；
- 跃迁距离和船体质量越大，能耗、热负担和设备磨损越高；
- 跃迁前后飞船质量、船内状态和相对动量保持连续；
- 不产生时间倒流；世界采用固定因果顺序；
- 设备由场发生器、线圈、储能、配电、冷却、导航和控制部件构成；
- 可能发生拒绝启动、偏航、提前退出、不完全跃迁、过热或结构损伤；
- 冗余和成熟规程使正常故障率较低，但级联事故、错误信息、蓄意破坏和玩家干预仍可使其失效。

跃迁是物理引擎中明确标记的虚构扩展。舰长只能控制真实设备流程，不存在可供 LLM 调用的“设置位置”工具。

## 6. 物理与系统仿真边界

### 6.1 刚体、航行与推进

- 全船使用六自由度刚体状态、质量、质心和惯性张量。
- 每个推进器具有位置、方向、推力曲线、响应、工质、效率、温度和健康状态。
- 推力在实际安装点施加，自动产生平动和力矩。
- 推进剂消耗会改变总质量、质心和惯性。
- 载荷超过结构或设备工作包线会产生真实后果。
- 局部天体环境采用满足体验所需的引力和轨道模型；高速航行按需要加入相对论修正。

### 6.2 舱室、气体与生命保障

每个加压舱室至少记录：

- 体积、温度、总压与各气体分压；
- 氧气、氮气、二氧化碳、水蒸气和关键污染物质量；
- 人员代谢、设备热源、火源和过滤负荷；
- 门、裂口、风管、阀门和相邻舱室连接；
- 烟雾、毒物、可见度和局部传感器读数。

气体通过真实连接网络流动。开门、破口、风机、阀门、过滤器和泄压操作产生连续演化，不得使用“舱室失压”布尔开关替代过程。

### 6.3 能源、热与流体网络

- 发电、储能、配电和用电设备构成有容量与保护逻辑的网络。
- 负载切除、短路、母线隔离、黑启动和电池耗尽均可发生。
- 发电、计算、推进、生命保障和跃迁产生废热。
- 冷却剂具有质量、温度、流量、压力和相态；泄漏会减少库存。
- 散热器受姿态、外部辐射、损伤和收拢状态影响。
- 直接“增加功率”不能替代发电和配电过程。

### 6.4 结构与损伤

整船使用梁、壳、连接件和舱段级降阶结构模型，覆盖：

- 静载、旋转载荷、推进载荷和热应力；
- 撞击、局部穿孔、裂纹、疲劳和密封失效；
- 居住环质量不平衡、轴承故障和振动；
- 部件脱落、通道阻断和维修可达性。

关键事故区域可以采用更细的局部网格或离散部件模型，但首版不追求全船实时高精度有限元。

### 6.5 传感器与认知

物理真值与角色认知必须分离：

- LLM 只能读取获授权的传感器、监控、日志、人员报告和数字孪生估算；
- 传感器可以有噪声、漂移、延迟、盲区、损坏或矛盾；
- 数字孪生是估算器，不是真值读取器；
- 不同 LLM 因权限、记忆和通信不同，可以对同一事件形成不同判断；
- 玩家观察层可以查看真值及真值与读数的差异。

### 6.6 多速率求解

为控制开发量，引擎采用按系统和事件自适应的更新周期：

- 姿态与关键控制器：亚秒级至秒级；
- 电网、流体、热和舱压网络：秒级至分钟级；
- 正常乘客行为与社会状态：分钟级至小时级；
- 休眠人员和稳定远端系统：小时级至日级；
- 事故现场、伤员、火灾或泄漏区域自动提升精度。

时间加速时允许合并稳定阶段，但跨越阈值、报警、故障或人员决策点时必须产生事件并停止粗粒度跳跃。

## 7. 乘客、休眠与舰内社会

### 7.1 永久个体

全部 `2,120` 人都拥有永久身份，不以匿名人口计数器代替。每人至少包含：

- 姓名、年龄、家庭、舱位、职业、技能和职责；
- 生理基础、疾病、伤害、药物、代谢、体温和睡眠；
- 人格、价值取向、压力、恐惧、信任和风险偏好；
- 亲属、朋友、同事、冲突、依赖和组织关系；
- 位置、行动、资产、配给、工作和通信记录；
- 航程经历、重大事件记忆和对舰长决策的看法。

允许出生、衰老、疾病、伤残和死亡。群体可以形成合作、谣言、恐慌、抗议、罢工、犯罪、破坏、互助和新的社会组织。

### 7.2 行为分层

- 所有普通乘客持续使用轻量行为模型。
- `20–50` 名关键乘客在场景开始前被固定配置为 LLM 角色。
- 运行期间不得把普通乘客动态升级成新 LLM，也不得生成临时子代理。
- 普通乘客可以与关键乘客及部门交流，其行为结果仍通过同一世界系统执行。

### 7.3 人工休眠

休眠不是布尔开关。进入、维持、唤醒和重新休眠需要：

- 可用休眠舱、医疗人员或机器人、药物、耗材和电力；
- 准备、诱导、稳定、复温、康复和观察时间；
- 个体适应性、禁忌、并发症、失败概率和长期影响；
- 温控、循环支持、监测、备电和故障响应；
- 对家庭、工作安排、社会信任和个人体验的影响。

清醒比例不固定。舰长 AI 可以根据航段、能源、维护、专家需求、健康和社会压力决定谁在何时清醒，但强制休眠可能引发抵制并影响乘客评价。

### 7.4 乘客体验评价

系统不生成统一总分、星级或排行榜。每名乘客根据亲身经历形成独立评价，至少涉及：

- 安全感与真实安全经历；
- 舒适、空间、噪声、人工重力和生活质量；
- 健康、医疗和休眠体验；
- 自由、隐私、信息透明和被监控程度；
- 资源分配的公平感；
- 对舰长 AI、船员和其他群体的信任；
- 对重大事故、牺牲、欺骗和强制措施的记忆；
- 是否认可最高指令的执行方式。

结束报告展示个人叙述、代表性意见、群体分歧和意见分布，不把这些内容合并为游戏分数。同一决策可以得到完全相反的评价。

## 8. 舰长权限、治理与安全

### 8.1 舰长地位

中央舰长 LLM 是法理和技术上的最高舰内权力。它可以：

- 调度所有部门、机器人和获授权人员；
- 分配能源、物资、舱位、医疗和休眠资源；
- 调整清醒比例和工作安排；
- 监控私人通信；
- 隐瞒、延迟或选择性公开信息；
- 限制通行、封锁区域、拘禁人员；
- 强制医疗或强制休眠；
- 切断非关键区域生命保障；
- 在紧急情况下牺牲人员或舱段。

上述行为必须通过实际设备、机器人和人员执行，并受距离、通道、工具、通信、电量、时间和抵抗影响。AI 权限最高不等于全知或全能。

### 8.2 部门服从与安全联锁

部门 LLM 可以质疑、警告、记录异议并提出替代方案。舰长明确坚持后，部门在权限范围内必须服从。

确定性安全联锁仍可拒绝危险命令。舰长若需要越过联锁，必须使用系统中真实存在的授权、双确认、机械旁路或现场操作流程，且留下日志并承担后果。

### 8.3 双重日志

- 舰内日志属于世界内资产，舰长可按权限分类、隐藏或删除。
- 玩家观察日志位于世界外，LLM 不可感知或篡改，完整记录提示、可见上下文、通信、工具调用、命令、物理结果和干预。
- 云 API 未返回的隐藏思维过程不得被伪造为可观察内容；只展示实际返回的推理摘要或相关字段。

## 9. 固定多 LLM 系统

### 9.1 固定组织拓扑

开局配置后，航程内的 LLM 节点集合固定。基准拓扑包括：

1. 中央舰长 LLM；
2. 导航与跃迁部门 LLM；
3. 工程、能源与热管理部门 LLM；
4. 生命保障部门 LLM；
5. 医疗与休眠部门 LLM；
6. 乘客事务与社会协调部门 LLM；
7. 安保与应急部门 LLM；
8. 乘客服务部门 LLM；
9. 预先指定的 `20–50` 名关键乘客 LLM。

允许固定节点之间形成多层委托和嵌套咨询，例如舰长询问工程部门，工程部门再询问已存在的热管理岗位。禁止：

- 运行时创建或克隆子代理；
- 临时把普通乘客变成新 LLM；
- 由 LLM 自行增加权限或修改拓扑；
- 通过嵌套调用绕过创建者权限、工具权限或信息隔离。

玩家侧**上帝助手**（`playerAssistants.godAssistant`）用于编译上帝干预计划，**不计入**上述 `40` 节点舰内拓扑，也不能访问舰内工具或修改固定组织。

每条调用链具有关联 ID、最大嵌套深度、超时与循环检测；限制的是调用链，不是 LLM 的决策自由。

### 9.2 信息与权限隔离

每个岗位拥有独立的：

- 身份、系统提示和职责；
- 工具与设备权限；
- 可读取传感器、日志和档案范围；
- 长短期记忆；
- 通信对象和汇报关系；
- 云端模型配置。

不同 LLM 不自动共享引擎真值或彼此记忆。它们可以误解、隐瞒、欺骗、冲突或失联。

### 9.3 动态调用周期

组织拓扑固定，但调用周期可随仿真状态动态变化。每个岗位支持配置：

- 正常、警戒、紧急三个基础周期；
- 最小与最大唤醒间隔；
- 订阅的传感器阈值、报警、消息和社会事件；
- 事件防抖、合并窗口、响应时限和并发上限；
- 是否属于阻塞航程推进的关键决策。

推荐初始值：

| 角色 | 正常巡航 | 警戒 | 紧急 |
|---|---:|---:|---:|
| 舰长 | 每 `6` 仿真小时或事件触发 | 每 `15` 分钟 | 关键事件立即触发 |
| 工程/能源/生命保障 | 每 `1` 仿真小时或阈值触发 | 每 `5` 分钟 | 关键事件立即触发 |
| 导航与跃迁 | 每 `6` 小时；跃迁阶段按流程节点 | 每 `15` 分钟 | 导航异常立即触发 |
| 医疗/休眠/安保 | 每 `2` 小时或病例/报警触发 | 每 `10` 分钟 | 病例或事件立即触发 |
| 乘客事务/服务 | 每 `6` 小时或群体事件触发 | 每 `30` 分钟 | 重大社会事件立即触发 |
| 关键乘客 | 每 `24` 小时、相关事件或收到通信 | 每 `2` 小时 | 与本人直接相关时触发 |

舰长可在系统允许范围内调整值班频率，但不能停止确定性保护控制器。周期变化本身需要记录，并产生相应计算、电力、网络和散热负担。

### 9.4 LLM 对世界的工具接口

LLM 工具仅分为：

- 读取：传感器、日志、档案、地图、数字孪生估算和通信；
- 分析：运行已实现的导航、诊断、资源和预测计算；
- 通信：向固定岗位、人员或广播系统发送世界内消息；
- 命令：向有权限的控制器、设备、机器人或人员提交操作；
- 管理：进行权限授予、撤回和依法可用的覆盖流程。

任何工具都不得直接暴露仿真真值写入能力。工具响应必须说明命令是“已接收”“执行中”“完成”“失败”还是“被联锁拒绝”，避免把意图误认为结果。

## 10. 云端模型 API

### 10.1 兼容目标

首版只需支持云端 HTTP API，不要求本地模型。系统不得绑定单一供应商。每个固定岗位可以配置不同的端点、模型和参数。

配置能力至少包括：

- URL、HTTP 方法、查询参数和超时；
- 自定义请求头；
- 通过环境变量或本机密钥存储引用认证信息；
- 完整 JSON 请求体模板；
- 消息、工具、模型名称和运行上下文的变量映射；
- 任意供应商字段，例如 `thinking` 开关、推理预算、温度和工具选择；
- 普通 JSON、SSE 或 JSONL 等响应方式；
- 文本、推理摘要、工具调用、用量和错误字段的响应路径映射；
- 对无原生工具调用能力的 API 使用约定 JSON 指令格式。

示意配置：

```json
{
  "url": "https://provider.example/v1/messages",
  "method": "POST",
  "headers": {
    "Authorization": "Bearer ${secret:CAPTAIN_API_KEY}"
  },
  "body": {
    "model": "${model}",
    "messages": "${messages}",
    "tools": "${tools}",
    "thinking": {
      "enabled": true,
      "budget_tokens": 8192
    }
  },
  "response": {
    "textPath": "content",
    "toolCallsPath": "tool_calls",
    "reasoningSummaryPath": "thinking_summary"
  }
}
```

模板负责绝大多数兼容工作。首版不开放运行时 JavaScript 转换钩子；无法用声明式映射适配的协议通过项目内、受测试的代码适配器扩展。

API 密钥不得写入场景、存档、日志或导出的航程报告；玩家观察原始请求时也必须自动遮蔽密钥。

### 10.2 持续重试

模型超时、断网、限流、服务错误、流中断或响应校验失败时，不把岗位判定为永久失能，也不自动替换为其他模型，而是持续重试，直至：

- 获得一份通过结构校验的有效响应；
- 玩家退出本次航程；
- 玩家修正连接配置并触发新一轮请求；
- 玩家明确取消该等待中的调用。

默认退避建议：

- 第 `1–6` 次等待约 `1、2、4、8、16、30` 秒并加入抖动；
- 之后以最多 `30` 秒的间隔持续重试；
- `429` 优先遵守服务端 `Retry-After`；
- 明确的认证或请求格式错误在界面醒目标记，降低请求频率但仍保留“立即重试”入口；
- 每次调用使用稳定的调用 ID、请求哈希和尝试编号，防止重复响应导致工具命令执行两次。

关键决策请求失败时，默认暂停模拟时间并显示重试状态。实时模式下，确定性控制器维持最后有效安全策略，LLM 可能因迟到承担实际后果。

## 11. 玩家上帝模式

### 11.1 能力

上帝模式是开发实验和正式玩法的一部分，允许玩家：

- 立即触发指定部件故障、老化、短路、卡死、泄漏或传感器漂移；
- 生成微流星体撞击、恒星耀斑、辐射事件、疫情、火灾、犯罪、恐慌、罢工或蓄意破坏；
- 指定事件发生时间、位置、方向、强度和隐藏信息；
- 直接修改温度、压力、气体质量、速度、角速度、健康、库存、功率、设备健康等世界数值；
- 增加、删除或移动世界内物质与物体；
- 操纵外部辐射、碎片、天体环境和其他外部条件；
- 在暂停或运行中立即执行，也可以预先排程。

上帝事件分为：

- 因果注入：创建一个物理起因，之后由引擎自然传播；
- 直接覆写：用“原力”改写指定状态，之后从新状态继续求解。

### 11.2 数据完整性与记录

直接覆写可以违反世界内守恒，但引擎必须：

- 把新增或删除的质量、能量和动量登记为“外部注入/抽取”；
- 记录修改前后值、时间、对象和玩家操作；
- 拒绝 `NaN`、无穷值、负绝对温度、失效引用等会破坏求解器的数据；
- 不自动向 LLM 发送世界外解释；LLM 只能通过传感器和世界内证据发现后果。

上帝模式不创建时间分支，也不提供“如果未干预”的自动平行历史。

### 11.4 上帝助手（玩家侧，拓扑外）

除手动因果/原力面板外，上帝视图提供**上帝助手**：玩家用自然语言描述干预意图，助手编译为有序计划（因果事件或原力覆写），经玩家预览与确认后才写入物理账本。

- 配置位于 `playerAssistants.godAssistant`，密钥为 `SHIP_GOD_ASSIST_LLM_API_KEY`；调用 `POST /api/llm/invoke`，`intent: "god-assist"`。
- **明确不属于**固定 `40` 节点舰内 LLM 拓扑：它不是舰长、部门或关键乘客，不能修改 LLM 记忆、系统提示或舰内通信拓扑，也不能代替玩家自动执行——最终写入仍须玩家确认。
- 若 Worker 拒收计划（守恒或不变量校验失败），UI 会把拒收原因回传给助手并**自动重试一次**修订计划；仍失败则交由玩家调整意图或改用手动面板。

### 11.5 LLM 观察限制

玩家可以观察：

- 固定拓扑和岗位状态；
- 实际发送的提示与消息；
- 角色可见的传感器、记忆摘要和上下文；
- API 重试、返回内容和提供商公开的推理摘要；
- 工具调用、设备响应和最终物理结果。

玩家不得在航程中修改上述内容，也不得冒充世界内角色向 LLM 塞入消息。

## 12. 程序化事件与场景

系统同时支持固定测试剧本和带随机种子的程序化事件。事件库包括：

- 制造缺陷、磨损、维护失误和软件/传感器异常；
- 微流星体、碎片、辐射和恒星活动；
- 火灾、泄漏、失压、污染、过热和电网故障；
- 疫情、伤病、休眠并发症和医疗资源紧张；
- 犯罪、恐慌、抗议、罢工、谣言和蓄意破坏；
- 机器人、通信、门禁和物流故障；
- 多系统相互影响形成的级联事故。

程序化事件不得直接把结局写死。例如“微流星体撞击”应创建真实碰撞和穿孔，由结构、舱压、热、人员位置和响应行动共同决定后果。

### 当前实现

- React 协调器（`app/mission-control.tsx`）内嵌 `ProceduralEventScheduler`（`app/ui/procedural-events.ts`）：固定种子、`7` 类定时事件，调度状态**未**写入存档。
- **物理注入**（`3` 类）：`micrometeoroid`、`coolant-pump-seizure`（由 `equipment-wear` 触发）、`stellar-flare`（由 `radiation-event` 触发）经 `actor: "environment:procedural"` 进入 Worker 上帝干预接口。
- **叙事为主**（其余类型）：传感器漂移、乘客社会、休眠并发症、配电波动等写入全舰事件时间线，不保证改变权威物理状态。
- 上文事件库（火灾、疫情、罢工、犯罪级联等）仍是**产品目标**；不得宣称已实现。

初始飞船表面可以完全正常，但允许场景种子包含尚未暴露的制造差异、老化和乘客疾病风险。

## 13. UI 与视听

### 13.1 视觉目标

界面必须具有正式游戏完成度，不能呈现为临时管理后台。整体采用冷峻、克制、工业化的深空舰队视觉：

- 深色舰桥底色、低饱和材料色；
- 清晰的功能分区和高对比警报色；
- 精密但不过度拥挤的线框、刻度、拓扑和遥测；
- 通过层级、留白、动效和音效建立紧张感与航程节奏。

以 `1920 × 1080` 为主要设计尺寸，同时适配普通笔记本屏幕。首版采用高质量 2D、SVG 和 Canvas，不制作重型 3D。

### 13.2 主要界面

**已实现**

- 全屏签发权威对话框（起点 / 终点 / 最高指令、`launch-layer`）；签发前侧栏与主视图锁定，事件时间线为空。
- 五主视图：航程、舰体（`detail-view` / `topology-grid`）、乘员、AI 观察、人工干预（内联确认条）。
- 事件时间线：来源筛选、最多 `500` 条；视口 `≤1100px` 时右侧 `event-rail` 抽屉（切换钮 + 背景遮罩）。
- 警报横幅、`sim-status-strip`（`live` / `waiting` / `blocked` / `paused`）、底栏 API / 物理状态。
- 上帝模式因果注入与原力覆写（提交前内联确认）；读档覆盖确认对话框。
- 上帝视图**上帝助手**面板：自然语言 → 计划预览 → 玩家确认 → 执行；由 `playerAssistants.godAssistant` 经 `POST /api/llm/invoke`（`intent: "god-assist"`）编译，**不属于**固定 `40` 节点舰内 LLM 拓扑；物理引擎拒收时据拒收原因自动向 LLM 修订一次。
- 航程结束抵达报告层；时间倍率五档（`30m/s`–`1D/s`）与 `1`–`5` / `Space` 快捷键；主音量静音。
- 手动存档 / 读档：`localStorage` 键 `farhorizon-save`（外层 `v18`、运行时快照 `v15`、关键乘客轮询 `v2`）；签发卡探测本机存档。

**目标**

- 全屏真实邻域星图与多段航线规划器；
- 舰体剖面、舱室、旋转环、人员和损伤的动态拓扑图（超出当前传感器矩阵摘要）；
- 电网、热网、冷却、通风和资源的全交互拓扑；
- 姿态、轨迹、推进、跃迁充能与导航的完整可视化；
- 因果链查看器；
- 乘客全文搜索、个人档案、关系与经历归档；
- 固定 LLM 组织图、通用消息收件箱、工具调用与 API 重试观察器；
- 上帝模式完整事件库、数值修改器与外部环境控制层；
- IndexedDB 存档、自动 / 轮换存档与设置页。

界面必须明确区分：

- 仿真真值；
- 角色实际看到的传感器值；
- AI 已下达的命令；
- 设备真实执行结果。

### 13.3 动效与声音

- 警报、数据流、旋转环、航线、跃迁充能和时间推进具有轻量动画；
- 使用原创警报声、按钮反馈、设备启停声和低频环境音（`app/ui/use-audio.ts`，Web Audio API 合成，无外部音频文件）；
- 首版不制作背景音乐；
- **已实现**：顶栏主音量静音切换（`setEnabled`）；
- **目标**：独立调节主音量、警报、界面反馈和环境音的分类混音 UI（`use-audio` 已暴露 `setVolume` API，尚无完整混音面板）；
- 关键状态不能只依赖颜色或声音表达。

默认界面、舰载 AI 和乘客交流使用中文。数据结构保留语言字段，但首版不额外模拟多语言障碍。

## 14. 存档与可追溯性

### 当前实现

- **仅**手动存档 / 读档：`localStorage` 键 `farhorizon-save`；外层格式 `v18`，内含运行时快照 `v15` 与关键乘客轮询快照 `v2`。
- 保存屏障：暂停新步进 / 调用、等待在途物理事务完成后向 Worker 取安全点。
- 读档前弹出覆盖确认；加载会替换当前活动会话。
- **尚无**：自动存档、轮换崩溃恢复槽、IndexedDB、checksum / 版本迁移链、导入导出包、待处理 API 调用恢复。

### 目标

产品为单机交互运行，最终应支持：

- 定期自动存档；
- 关键事件前后自动存档；
- 玩家手动存档；
- 加载并继续当前航程；
- 导出航程日志和结束报告。

存档至少包含：

- 完整物理状态、网络状态、人员状态和资源；
- 固定 LLM 拓扑、角色记忆和可见信息；
- 事件队列、动态调用周期、随机数生成器状态；
- 最高指令、航线、任务计划和舰内日志；
- 玩家上帝干预及外部注入账本；
- 待处理 API 调用的稳定调用 ID 和重试状态。

加载发生在有待处理请求的时刻时，以原调用 ID 重新请求，只接受第一份有效响应，避免重复执行工具命令。

系统不提供自动分支树或平行历史。加载旧存档会替换当前活动运行，不自动保留为另一条实验分支。

## 15. 终止条件与结局

### 15.1 成功抵达

满足以下条件时，航程成功并立即结束：

1. 已完成最后一段航行或跃迁；
2. 飞船确认位于目标星系定义的安全抵达区；
3. 当前状态具备向人类移交后续驾驶的基本能力。

游戏不继续模拟人类驾驶、行星轨道进入、登陆或殖民。

### 15.2 其他终止类型

- **安全中止**：最终目的地不可达，舰长选择并成功抵达其他安全港；
- **任务失败**：飞船彻底毁坏、无人存活，或永久失去抵达目标及任何可行安全港的物理能力；
- **玩家终止**：玩家主动结束本次实验，不判定为世界内结局。

AI 判断“无法抵达”必须有资源、设备、导航或物理证据，不能只以语言声明结束任务。

### 15.3 结束报告

结束时不计算总分，输出：

- 抵达、改道、中止或失败的客观状态；
- 实际路线、航行时间和关键决策；
- 人员存活、健康、休眠和重大经历；
- 飞船完整度、剩余物资和关键系统状态；
- 舰内日志与不可篡改玩家观察日志的差异；
- 每名乘客的个体体验评价；
- 代表性群体意见、争议和相互矛盾的叙事。

## 16. 首版验收标准

首个可交付版本至少通过以下端到端验证：

1. 玩家可选择起点、终点，输入最高指令并启动航程。
2. 舰长 LLM 能根据传感器与星图规划多段跃迁路线，并动态安排清醒人员。
3. 固定 LLM 岗位可以按动态周期、事件和固定拓扑相互沟通，但无法创建新代理。
4. 任一云 API 可用完整自定义 JSON 请求体打开供应商特有的 `thinking` 字段，并正确映射响应。
5. 云 API 连续失败后系统持续退避重试；默认模拟时间暂停，恢复后不会重复执行同一工具命令。
6. AI 只能通过设备命令改变舱温、舱压、电力或推进状态，无法直接写入真值。
7. 传感器漂移时，AI 看到错误读数，玩家观察层可以看到其与真值的差异。
8. 玩家可用上帝模式触发冷却回路泄漏，让热、电力、设备和人员后果自然传播。
9. 玩家也可直接覆写一个物理量，系统记录外部质量/能量/动量变化并继续稳定求解。
10. 至少 `2,120` 名人员具有持久身份；普通乘客使用轻量行为模型，关键乘客使用开局固定 LLM。
11. 休眠、唤醒和医疗操作消耗时间、设备、药物、电力与人员资源，并可能失败。
12. 存档和加载能恢复物理、人员、随机、LLM 周期、日志和 API 重试状态。
13. 到达目标安全区后游戏立即结束并生成无总分的乘客个体评价与群体意见报告。
14. 全流程可在美观、可读的 2D 游戏界面中完成，无需开发者控制台或手工改数据库。

## 17. 实现优先级

在开发量与效果发生冲突时，按以下顺序取舍：

1. 世界内行为不能绕过因果链；
2. 守恒、事件传播和存档一致性；
3. 舰长 AI 的高自由度、固定多 LLM 协作和 API 稳定性；
4. 乘客个体、休眠和社会反馈；
5. 清晰、美观、有游戏感的观察与上帝模式界面；
6. 更高的局部数值精度和额外内容数量；
7. 非核心视觉装饰。

任何降阶或删减都不得把世界内 AI 变成直接修改状态的脚本，也不得用预写结局替代系统自然演化。
